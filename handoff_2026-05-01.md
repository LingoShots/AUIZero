## Important Pilot Feedback To Fix Next

The next coding session should prioritize second-round pilot bugs before more refactoring.

### Critical Bug 1: Assignment Not Publishing Correctly

- A reported issue from pilot testing says assignments may not always publish correctly.
- Next session should trace the full publish path:
  - teacher clicks publish
  - assignment status updates in Supabase
  - student assignment list reloads by class
  - published assignment appears without requiring fragile refresh behavior
  - email notification is sent only after a confirmed publish
- Add defensive UI messaging if publish fails.
- Do not show a publish success message unless the server confirms the assignment is actually published.

### Critical Bug 2: Rubric Criteria Count / Total Score Mismatch Blocks Submission

- Reported pilot bug: a rubric parsed as 3 criteria accidentally instead of 4.
- The app still treated the rubric total as 20 instead of 15, apparently calculating as 5 x 4 instead of 5 x 3.
- This mismatch stopped students from submitting their work.
- This must be fixed in a student-safe way:
  - never block final submission because of rubric metadata mismatch
  - preserve draft/final/reflection/self-assessment work even if submit fails
  - show a clear error near the submit button if submission fails
  - let students return later and submit once the teacher/app issue is fixed
  - reconcile rubric totals from actual parsed criteria instead of stale expected criteria count

Suggested investigation points:

- Check rubric schema parsing output: criteria length, `totalPoints`, `maxScore`, row scores.
- Check student self-assessment requirements on final submit.
- Check whether submission validation requires all rubric rows to be selected when parsed criteria count does not match total points.
- Check whether teacher-uploaded rubric totals are derived from uploaded schema or overwritten by assignment total points.
- Add a fallback that recalculates display/submission total from parsed criteria when criteria exist.

### Data Preservation Requirement

- The highest priority is that students must never lose work.
- If any submit/save path fails:
  - keep all student work in local backup
  - retry server sync where possible
  - prevent the UI from pretending submission succeeded
  - keep the assignment selectable so the student can come back and submit
